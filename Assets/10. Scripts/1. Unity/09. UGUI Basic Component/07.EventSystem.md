# 모든 UI 입력을 관장하는, Event System

## Event System이란?

거대한 콘서트홀의 음향 감독을 상상해봅시다. 무대 위에는 수많은 마이크(가수, 악기 등)가 있고, 관객석에서는 누군가 박수를 치거나 환호성을 지릅니다. 음향 감독은 이 모든 '소리(이벤트)'들을 입력받아, 어떤 소리가 중요한지 판단하고, 그에 맞는 소리를 앰프와 스피커로 내보내는 역할을 합니다. 만약 음향 감독이 없다면, 어떤 소리도 처리되지 못하고 관객에게 전달되지 않을 것입니다.

유니티 UI에서 **Event System(이벤트 시스템)** 은 바로 이 '음향 감독'과 같습니다. 사용자가 마우스를 클릭하거나, 키보드를 누르거나, 화면을 터치하는 등의 모든 입력 '이벤트'를 감지하고, 이 이벤트가 어떤 UI 요소(버튼, 슬라이더 등)에서 발생했는지를 판단하여 해당 UI 요소에게 "너 클릭됐어!"라고 알려주는 핵심적인 역할을 합니다.

정의하자면, `Event System`은 사용자의 입력을 받아 처리하고, 이를 적절한 게임 오브젝트에게 전달하는 UGUI의 중앙 허브입니다. `Canvas`를 생성할 때 자동으로 함께 생성되며, **이벤트 시스템이 없으면 버튼 클릭, 슬라이더 조작 등 UI에 대한 어떤 상호작용도 작동하지 않습니다.**

## 핵심 요약

-   Event System은 **모든 사용자 입력(클릭, 터치, 키보드 등)을 감지하고 처리**하는 중앙 관리자입니다.
-   `Canvas`를 만들 때 자동으로 생성되며, 씬(Scene)에 **단 하나만 존재**해야 합니다.
-   **Input Module**을 통해 어떤 종류의 입력을 받을지 결정합니다. (예: 마우스/키보드, 터치)
-   **Raycaster**를 통해 입력이 발생한 위치에 어떤 UI 요소가 있는지를 감지합니다.

## 세부 개념

`Event System`은 그 자체로 하나의 오브젝트이며, 여러 개의 중요한 컴포넌트가 조합되어 작동합니다.

### 1. Event System 컴포넌트

이벤트 시스템의 동작을 관리하는 메인 컴포넌트입니다.

**Public Field**

| 항목 | 설명 |
| :--- | :--- |
| `First Selected` | 씬이 시작될 때 또는 이 이벤트 시스템이 활성화될 때 기본적으로 선택될 게임 오브젝트를 지정합니다. 주로 키보드나 게임패드 조작을 위한 첫 포커스 설정에 사용됩니다. |
| `Send Navigation Events` | 키보드나 게임패드를 이용한 UI 네비게이션 이벤트를 사용할지 여부를 결정합니다. |

### 2. Input Module (입력 모듈)

어떤 종류의 하드웨어 입력을 처리할지를 결정하는 컴포넌트입니다. Unity 2021 이후부터는 새로운 `Input System` 패키지 기반의 `Input System UI Input Module`이 기본으로 사용됩니다.

-   **Input System UI Input Module (최신)**: 새로운 `Input System`과 연동하여 작동합니다. 마우스, 키보드, 게임패드, 터치 등 다양한 장치의 입력을 유연하게 처리할 수 있습니다. `Actions Asset`을 통해 입력 액션을 커스터마이징할 수 있습니다.
-   **Standalone Input Module (구형)**: 전통적인 `Input.GetAxis`, `Input.GetMouseButtonDown` 등 레거시 입력 방식을 사용합니다. PC 환경의 마우스 및 키보드 입력을 처리합니다.

### 3. Raycaster (레이캐스터)

'광선(Ray)을 쏘는(cast) 장치'라는 뜻으로, 화면의 특정 좌표(마우스 커서, 터치 지점)에서 보이지 않는 광선을 쏴서 어떤 UI 요소와 충돌하는지를 감지하는 매우 중요한 컴포넌트입니다.

-   **Graphic Raycaster**: `Canvas`에 부착되어 있으며, 해당 `Canvas` 내의 모든 UI 그래픽 요소들을 대상으로 레이캐스팅을 수행합니다. `Button`의 `Image`나 `Text` 등이 모두 레이캐스팅의 대상이 됩니다.
    -   `Blocking Objects`와 `Blocking Mask` 속성을 통해 특정 2D/3D 물리 오브젝트가 UI로 향하는 레이캐스트를 막도록 설정할 수 있습니다.
-   **Physics Raycaster**: 3D 물리 오브젝트(콜라이더가 있는)를 대상으로 레이캐스팅을 수행합니다. 3D 오브젝트를 클릭하는 것과 같은 상호작용을 UI 이벤트 시스템과 통합할 때 카메라에 부착하여 사용합니다.
-   **Physics 2D Raycaster**: 2D 물리 오브젝트(2D 콜라이더가 있는)를 대상으로 레이캐스팅을 수행합니다.

**이벤트 처리 과정**
1.  사용자가 화면을 클릭합니다.
2.  `Input Module`이 이 클릭 입력을 감지합니다.
3.  `Event System`은 씬에 있는 모든 `Raycaster`에게 "이 좌표에 누가 있는지 확인해!"라고 명령합니다.
4.  `Graphic Raycaster`는 자신의 `Canvas` 내 UI 요소들을 검사하고, `Physics Raycaster`는 3D 오브젝트들을 검사합니다.
5.  레이캐스트에 가장 먼저 감지된(가장 위에 있는) `GameObject`를 찾아냅니다.
6.  `Event System`은 해당 `GameObject`에게 `IPointerClickHandler`와 같은 이벤트 인터페이스를 통해 "클릭 이벤트가 발생했다"고 메시지를 전달합니다.
7.  `Button` 컴포넌트는 이 메시지를 받아 자신의 `On Click()` 이벤트를 실행합니다.

## 코드 예제

`Event System`은 보통 직접 스크립트로 제어하기보다는, `Event System`이 보내주는 이벤트를 받는 쪽을 구현하는 경우가 대부분입니다. 아래는 `Event System`이 제공하는 다양한 이벤트 인터페이스를 활용하는 예제입니다.

```csharp
// CustomButtonHandler.cs
// 버튼의 OnClick 외에 마우스가 들어오고 나가는 등의 세부 이벤트를 처리하는 스크립트

using UnityEngine;
using UnityEngine.EventSystems; // 이벤트 인터페이스를 사용하기 위해 필수

// 스크립트가 부착된 오브젝트가 해당 인터페이스의 이벤트를 받을 수 있게 됨
public class CustomButtonHandler : MonoBehaviour, IPointerEnterHandler, IPointerExitHandler, IPointerDownHandler, IPointerUpHandler
{
    // 마우스 커서가 이 오브젝트 영역으로 들어왔을 때 호출
    public void OnPointerEnter(PointerEventData eventData)
    {
        Debug.Log("마우스가 들어왔습니다!");
        // 예: 버튼이 커지는 애니메이션 재생
    }

    // 마우스 커서가 이 오브젝트 영역에서 나갔을 때 호출
    public void OnPointerExit(PointerEventData eventData)
    {
        Debug.Log("마우스가 나갔습니다!");
        // 예: 버튼이 원래 크기로 돌아가는 애니메이션 재생
    }

    // 이 오브젝트 위에서 마우스 버튼을 눌렀을 때 호출
    public void OnPointerDown(PointerEventData eventData)
    {
        Debug.Log("마우스를 누르는 중...");
    }

    // 이 오브젝트 위에서 눌렀던 마우스 버튼을 뗐을 때 호출 (클릭!)
    public void OnPointerUp(PointerEventData eventData)
    {
        Debug.Log("마우스를 뗐습니다!");
    }
}
```
*위 스크립트를 `Image`나 `Button`이 있는 `GameObject`에 추가하면, `Event System`이 마우스 움직임을 감지하여 각 상황에 맞는 함수를 자동으로 호출해줍니다.*

## Note: 주의사항

-   **씬에 단 하나만**: `Event System`은 씬에 두 개 이상 존재하면 안 됩니다. 입력 처리에 혼란이 생겨 UI가 오작동하는 원인이 됩니다. 씬을 넘나들며 계속 사용해야 한다면 `DontDestroyOnLoad`를 적용하여 하나만 유지되도록 관리해야 합니다.
-   **`Event Camera` 할당**: `World Space` 렌더 모드의 `Canvas`는 `Event Camera` 속성에 카메라를 직접 지정해주어야 `Event System`이 해당 `Canvas`의 UI를 정상적으로 감지하고 상호작용할 수 있습니다.
-   **`Raycast Target`과 성능**: `Image`나 `Text`의 `Raycast Target` 옵션은 `Graphic Raycaster`의 검사 대상이 될지 여부를 결정합니다. 상호작용이 필요 없는 모든 장식용 그래픽은 이 옵션을 꺼야 불필요한 레이캐스트 검사를 줄여 성능을 향상시킬 수 있습니다.

## Further Study: 확장 학습

-   **이벤트 인터페이스 (Event Interfaces)**: `IPointerClickHandler`, `IBeginDragHandler`, `IDragHandler`, `IEndDragHandler`, `IScrollHandler` 등 `Event System`이 제공하는 다양한 인터페이스에 대해 학습해보세요. 이를 통해 드래그 앤 드롭, 스크롤 등 복잡한 커스텀 상호작용을 구현할 수 있습니다.
-   **`ExecuteEvents` 클래스**: 스크립트 코드에서 특정 `GameObject`에게 강제로 이벤트를 발생시키고 싶을 때 `ExecuteEvents` 클래스를 사용할 수 있습니다. 예를 들어, `ExecuteEvents.Execute(myButton, new PointerEventData(EventSystem.current), ExecuteEvents.pointerClickHandler);` 코드는 `myButton`을 강제로 클릭한 것처럼 만듭니다.
-   **커스텀 입력 모듈 (Custom Input Module)**: `BaseInputModule`을 상속받아 자신만의 입력 처리 방식을 가진 모듈을 만들 수 있습니다. 예를 들어, 자이로 센서나 음성 인식을 UI 입력으로 사용하는 등의 특수한 상호작용을 구현할 수 있습니다.
