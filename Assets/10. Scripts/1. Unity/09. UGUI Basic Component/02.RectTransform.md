# RectTransform이란 무엇인가?

여러분이 화이트보드에 포스트잇 메모를 붙여 정리하는 상황을 상상해봅시다.
각각의 메모는 저마다의 위치와 크기를 가집니다. 그리고 여러분은 메모의 모서리를 
보드 가장자리에 맞춰 '고정'할 수도 있습니다. 만약 누군가 화이트보드의 크기를 
양옆으로 늘린다면, 여러분이 어떻게 메모를 붙여두었는지에 따라 메모가 함께 늘어나거나,
혹은 구석에 그대로 머물러 있을 겁니다.

유니티에서 **RectTransform(렉트 트랜스폼)** 은 바로 이 포스트잇 메모를 다루는 
방식과 같습니다. UI라는 메모지를 Canvas라는 화이트보드 위에 어떻게 배치하고, 
'핀(앵커)'을 어디에 꽂아 크기 변화에 대응할지를 결정하는 시스템입니다.

정의하자면, `RectTransform`은 모든 UI 요소가 위치, 크기, 그리고 부모에 대한 
정렬 방식을 정의하기 위해 사용하는 핵심 컴포넌트입니다. 3D 오브젝트의 `Transform` 
컴포넌트를 2D UI 레이아웃에 맞게 확장한 버전으로, UI 배치를 위한 강력하고 
필수적인 기능들을 제공합니다.

## 핵심 요약

-   `RectTransform`은 `Transform`을 상속받아 UI 요소를 위한 **사각형 형태의 레이아웃 정보**를 추가로 가집니다.
-   **Pivot(피벗)**: UI 사각형의 중심점입니다. 위치, 크기 및 회전의 기준이 됩니다. (0,0)은 좌측 하단, (0.5, 0.5)는 정중앙, (1,1)은 우측 상단을 의미합니다.
-   **Anchors(앵커)**: 부모 `RectTransform`에 UI 요소를 '고정'하는 네 개의 기준점입니다. 부모의 크기가 변할 때 UI 요소가 어떻게 반응할지를 결정하는 **가장 중요한 속성**입니다.
-   앵커가 한 점에 모여있으면 UI는 **고정 크기**를 유지하고, 앵커가 분리되어 있으면 부모와의 **상대적인 간격**을 유지하며 함께 늘어납니다.

## 세부 개념

### 1. Pivot (피벗)

피벗은 `RectTransform` 사각형 내의 기준점으로, 모든 변형(위치, 회전, 크기 조절)이 이 점을 중심으로 일어납니다. X, Y 각각 0에서 1 사이의 정규화된 값으로 표현됩니다. 예를 들어, `(0.5, 0.5)`는 사각형의 정중앙을, `(0, 1)`은 좌측 상단 모서리를 의미합니다. 우리가 UI 요소의 위치를 지정할 때, 실제로는 그 UI의 피벗 지점의 위치를 지정하는 것입니다.

**Public Field**

| 항목 | 설명 |
| :--- | :--- |
| `Pivot` | RectTransform의 회전 및 크기 조절의 중심점입니다. X, Y 각각 0~1 사이의 값으로 표현됩니다. (0, 0)은 좌하단, (0.5, 0.5)는 중앙, (1, 1)은 우상단입니다. |

-   **활용 예시**: 오른쪽에서 왼쪽으로 줄어드는 체력 바를 만든다고 가정해봅시다. 체력 바 Image의 `Pivot`을 `(0, 0.5)` (왼쪽 중앙)로 설정합니다. 이제 스크립트에서 `RectTransform`의 너비(`width`)를 줄이면, 피벗이 왼쪽에 있으므로 사각형이 왼쪽을 기준으로 오른쪽 방향으로 줄어들게 됩니다. 만약 피벗이 중앙(`0.5, 0.5`)이었다면 양쪽에서 중앙을 향해 동시에 줄어들었을 것입니다.

### 2. Anchors (앵커)

앵커는 UI 요소가 부모 요소에 어떻게 '연결'될지를 결정하는 가장 중요하고 핵심적인 개념입니다. 앵커가 설정된 방식에 따라, 부모의 크기가 변할 때 자식 UI가 크기를 고정할지, 혹은 함께 늘어날지가 결정됩니다.

#### 1) 앵커가 한 점에 모여 있을 때 (고정 크기 모드)

네 개의 앵커가 한 지점에 뭉쳐 있으면, UI 요소는 부모의 크기 변화에 관계없이 항상 일정한 너비와 높이를 유지합니다. 위치는 앵커 지점을 기준으로 얼마나 떨어져 있는지를 나타내는 `Anchored Position` 값으로 결정됩니다. 아이콘, 고정 크기 버튼처럼 크기가 변하면 안 되는 요소에 사용됩니다.

**Public Field**

| 항목 | 설명 |
| :--- | :--- |
| `Anchored Position` | 앵커로부터 피벗까지의 상대적인 위치(오프셋)입니다. |
| `Width`, `Height` | UI 요소의 고정 너비와 높이입니다. |

-   **활용 예시**: 화면 우측 상단에 항상 고정되어 있는 50x50 크기의 '설정' 아이콘을 배치할 때. 인스펙터의 앵커 프리셋에서 'top-right'를 선택하면 앵커가 우측 상단으로 모입니다. 이제 화면 해상도가 어떻게 변하든 아이콘은 항상 우측 상단 모서리로부터 일정한 거리를 유지하며 크기는 변하지 않습니다.

#### 2) 앵커가 분리되어 있을 때 (스트레칭 모드)

네 개의 앵커가 서로 다른 위치에 떨어져 있으면, UI 요소는 앵커들이 형성하는 사각형 영역을 기준으로 부모의 크기 변화에 맞춰 함께 늘어나거나 줄어듭니다. 이때 UI 요소의 사방 경계는 각 앵커로부터의 고정된 간격(Margin)을 유지하게 됩니다.

**Public Field**

| 항목 | 설명 |
| :--- | :--- |
| `Left`, `Top` | 각각 부모의 좌측, 상단 경계(앵커)로부터 UI 요소의 좌측, 상단 경계까지의 거리입니다. |
| `Right`, `Bottom` | 각각 부모의 우측, 하단 경계(앵커)로부터 UI 요소의 우측, 하단 경계까지의 거리입니다. |

-   **활용 예시**: 화면 전체 너비에 맞춰져야 하는 상단 알림 바를 만들 때. 앵커 프리셋에서 'top-stretch'를 선택합니다. 이렇게 하면 상단 앵커 2개는 부모의 상단에, 하단 앵커 2개는 조금 아래에 위치하며 좌우로 벌어집니다. 이제 `Left`와 `Right` 값을 0으로 설정하면 화면 너비에 꽉 차게 되고, `Top`과 `Bottom` 값으로 높이를 조절할 수 있습니다. 화면 너비가 변하면 알림 바도 함께 변합니다.
-   **잘못된 사용 예시**: 앵커가 분리된 스트레칭 모드에서 인스펙터를 통해 `Width`나 `Height`를 직접 수정하려고 하는 경우. 이 값들은 비활성화되어 있습니다. 왜냐하면 크기가 앵커와 `Left`, `Right`, `Top`, `Bottom` 값에 의해 자동으로 계산되기 때문입니다.

## 코드 예제
```csharp
// NotificationSlider.cs
// 화면 상단에 나타났다가 사라지는 알림 패널을 제어하는 스크립트

using UnityEngine;
using System.Collections;

public class NotificationSlider : MonoBehaviour
{
    private RectTransform panelRect;
    private Vector2 onScreenPosition;
    private Vector2 offScreenPosition;

    public float slideSpeed = 3.0f;
    public bool isShown = false;

    void Start()
    {
        panelRect = GetComponent<RectTransform>();
        
        // 이 패널의 앵커는 top-stretch로 가정 (Y앵커는 1)
        // anchoredPosition.y = 0 이 앵커 위치(화면 상단)를 의미
        float panelHeight = panelRect.rect.height;

        // 화면 상단에 보일 때의 위치 (패널 높이의 절반만큼만 내림)
        onScreenPosition = new Vector2(0f, -panelHeight / 2);
        // 화면 상단 밖에 숨겨질 때의 위치 (패널 높이의 절반만큼 올림)
        offScreenPosition = new Vector2(0f, panelHeight / 2);

        // 시작 시 화면 밖에 위치
        panelRect.anchoredPosition = offScreenPosition;
    }

    // 버튼 등에서 이 함수를 호출하여 패널을 토글
    public void TogglePanel()
    {
        isShown = !isShown;
        Vector2 targetPos = isShown ? onScreenPosition : offScreenPosition;
        
        // 이전에 실행중인 코루틴이 있다면 중지
        StopAllCoroutines();
        StartCoroutine(SlideTo(targetPos));
    }

    IEnumerator SlideTo(Vector2 target)
    {
        while (Vector2.Distance(panelRect.anchoredPosition, target) > 0.1f)
        {
            panelRect.anchoredPosition = Vector2.Lerp(panelRect.anchoredPosition, target, Time.deltaTime * slideSpeed);
            yield return null;
        }
        // 정확한 위치로 보정
        panelRect.anchoredPosition = target;
    }
}
```

## 주요 Public 속성(Property) 및 메서드

| 속성/메서드 | 타입 | 설명 |
| :--- | :--- | :--- |
| `anchoredPosition` | `Vector2` | 앵커를 기준으로 한 피벗의 위치입니다. 앵커가 모여있을 때 주로 사용합니다. |
| `sizeDelta` | `Vector2` | 앵커가 모여있을 때는 사각형의 크기(너비, 높이)를 의미하고, 앵커가 떨어져 있을 때는 앵커가 정의하는 영역 크기에 대한 가감 수치입니다. |
| `offsetMin` | `Vector2` | 사각형의 좌측 하단 코너가 좌측 하단 앵커로부터 얼마나 떨어져 있는지를 나타냅니다. 스크립트에서 `(Left, Bottom)` 값을 제어할 때 사용합니다. |
| `offsetMax` | `Vector2` | 사각형의 우측 상단 코너가 우측 상단 앵커로부터 얼마나 떨어져 있는지를 나타냅니다. 스크립트에서 `(-Right, -Top)` 값을 제어할 때 사용합니다. |
| `rect` | `Rect` | 로컬 공간(Local Space)에서의 사각형 위치와 크기를 담고 있는 `Rect` 구조체입니다. (읽기 전용) |
| `SetSizeWithCurrentAnchors()` | `void` | 현재 앵커 설정을 기준으로 `RectTransform`의 크기를 변경하는 메서드입니다. |
| `GetWorldCorners()` | `void` | 사각형의 네 꼭짓점 좌표를 월드 공간(World Space) 기준으로 계산하여 배열에 채워줍니다. UI 요소가 화면 밖에 있는지 판별할 때 유용합니다. |

## 활용 예시

`RectTransform`은 `Layout Group` 컴포넌트들과 함께 사용할 때 진정한 강력함을 발휘합니다. 화면 크기에 따라 아이템 슬롯 개수가 변하는 인벤토리를 만든다고 가정해봅시다.

1.  인벤토리 배경이 될 `Panel`을 만들고, 앵커를 `stretch-stretch` 프리셋으로 설정하여 화면을 채우되 약간의 여백을 줍니다.
2.  이 `Panel`에 `Grid Layout Group` 컴포넌트를 추가합니다. 이 컴포넌트는 자식 오브젝트들을 바둑판식 격자 형태로 자동 정렬해줍니다.
3.  인벤토리 아이템 슬롯으로 사용할 `Image` UI를 프리팹(Prefab)으로 만듭니다.
4.  스크립트를 작성하여, 인벤토리에 아이템이 추가될 때마다 슬롯 프리팹을 `Panel`의 자식으로 생성(`Instantiate`)합니다.
5.  결과: 우리는 슬롯의 `RectTransform`을 직접 제어하지 않았지만, 부모인 `Panel`의 `Grid Layout Group`이 모든 자식 슬롯들의 위치와 정렬을 자동으로 처리해줍니다. `Panel`의 `RectTransform`이 화면 크기에 맞춰 늘어나면, `Grid Layout Group`은 그에 맞춰 내부 슬롯들의 배치(예: 한 줄에 몇 개씩 보여줄지)를 동적으로 조절할 수 있습니다.

이처럼 `RectTransform`의 앵커 시스템과 레이아웃 컴포넌트를 조합하면 복잡하고 유연한 UI를 손쉽게 구현할 수 있습니다.

## Note: 주의사항

-   **스크립트 제어**: 스크립트로 `RectTransform`을 제어할 때는 `transform.position`이나 `transform.localPosition`을 직접 수정하는 것보다 `anchoredPosition`, `sizeDelta`, `offsetMin/Max` 등 `RectTransform`이 제공하는 전용 속성을 사용하는 것이 좋습니다. 그렇지 않으면 `Canvas`의 레이아웃 시스템이 값을 매 프레임 덮어써서 예기치 않은 결과가 발생할 수 있습니다.
-   **Layout Group과의 관계**: `Vertical Layout Group`, `Grid Layout Group` 같은 레이아웃 컴포넌트가 적용된 오브젝트의 자식들은 `RectTransform` 속성 대부분이 레이아웃 그룹에 의해 제어됩니다. 이 경우 인스펙터에서 값들이 비활성화되며, 스크립트로 수정해도 소용없습니다.
-   **`sizeDelta`의 의미**: `sizeDelta`는 앵커가 모여있을 때는 '크기'를 의미하지만, 앵커가 떨어져 있을 때는 '앵커 영역 대비 추가되거나 빠지는 여백의 합'을 의미하여 혼란을 줄 수 있습니다. 앵커의 상태에 따라 `sizeDelta`의 의미가 달라진다는 점을 항상 기억해야 합니다.

## Further Study: 확장 학습

-   **Layout Groups**: `Vertical/Horizontal Layout Group`, `Grid Layout Group` 등 UI 요소들을 자동으로 정렬해주는 강력한 레이아웃 컴포넌트들에 대해 깊이 학습해보세요.
-   **Content Size Fitter**: 자식 요소들의 크기에 맞춰 부모 `RectTransform`의 크기를 자동으로 조절해주는 컴포넌트입니다. 내용의 길이에 따라 크기가 변하는 텍스트 창이나 목록 표시에 매우 유용합니다.
-   **Auto Layout System**: 유니티의 UI 자동 레이아웃 시스템이 내부적으로 어떻게 계산되고 업데이트되는지(Layout Rebuild) 학습하면 복잡한 UI에서 발생하는 성능 문제를 진단하고 해결하는 데 큰 도움이 됩니다.
